# The Internals of CSS Layout — Box Model, Flexbox, Grid, and Positioning Algorithms

> *CSS layout is not "vibes" — it is a deterministic, multi-phase pipeline that takes a tree of computed styles and produces a tree of boxes with used values for `width`, `height`, `x`, and `y`. Understanding the algorithms that implement Flexbox, Grid, and the Box Model explains every "why is my layout broken?" you have ever asked. The browser does not guess; it follows §-numbered specs to the letter.*

---

## 1. The Visual Formatting Model — Overview

The Visual Formatting Model is the canonical layout algorithm defined in CSS 2.1 §9 and updated by CSS Display Level 3, Box Level 3, and the formatting-context modules. It defines what a *box* is, how boxes are arranged, and how their geometry is computed.

### 1.1 The Principal Box

Every element in the rendered document tree generates **zero or more boxes**. The default — for an element with `display: block`, `display: inline`, or `display: inline-block` — is **one principal box**, plus possibly **anonymous boxes** generated by the renderer to satisfy invariants of a formatting context.

```html
<div class="parent">
  Hello
  <p>World</p>
  Tail
</div>
```

The browser wraps the bare text nodes (`Hello`, `Tail`) in **anonymous block boxes** so that the principal box of `.parent` (which establishes a Block Formatting Context once `<p>` joins) contains only block-level children. Anonymous boxes inherit only inheritable properties; you cannot style them directly.

### 1.2 Block-Level vs Inline-Level

| Level | Default `display` | Layout role |
|:-----|:-----|:-----|
| Block-level | `block`, `list-item`, `table` | Stacks vertically, full-width by default, owns its line |
| Inline-level | `inline`, `inline-block`, `inline-table` | Flows horizontally inside a line box |
| Run-in | `run-in` | Joins next block as inline (rarely supported) |

A block-level box participates in a **Block Formatting Context (BFC)** — boxes stacked top-to-bottom (or in the block-axis direction). An inline-level box participates in an **Inline Formatting Context (IFC)** — boxes laid into line boxes.

### 1.3 The Layout Pass — What It Produces

Layout is the phase between style resolution and paint. Its job: take *computed values* (the property values after cascade + inheritance + initial-value resolution) and produce *used values*: concrete numbers for box geometry.

```
Computed values (style tree)
        ↓
Layout (formatting context algorithms)
        ↓
Used values (box tree with width, height, x, y)
        ↓
Paint (display lists, layers)
        ↓
Composite (GPU layer composition)
```

Used values are the answers to questions like "what is this `<div>`'s pixel width?" — `width: 50%` is a computed value; `width: 423.5px` is the used value after the parent's inline-size is resolved.

### 1.4 Spec Walkthrough — CSS 2.1 §9

The canonical Visual Formatting Model spec (CSS 2.1 §9.1) defines:

- §9.1 — Viewport, canvas, initial containing block
- §9.2 — Block-level vs inline-level boxes; BFC and IFC introduction
- §9.3 — Positioning schemes: normal flow, floats, absolute positioning
- §9.4 — Normal flow (BFC/IFC details)
- §9.5 — Floats (the float algorithm)
- §9.6 — Absolute positioning
- §9.7 — Relationships between display, position, and float
- §9.8 — Comparison of normal flow, floats, and absolute positioning
- §9.9 — Layered presentation (z-index, stacking contexts)
- §9.10 — Text direction

Modern modules (Flexbox, Grid, Display 3, Position 3) supersede or extend specific sections, but the §9 vocabulary — *containing block*, *formatting context*, *normal flow* — is the bedrock.

## 2. The Box Model — Computation

The Box Model defines the geometry of every box: a content area surrounded by padding, border, and margin.

```
+--------------------------------------+
|            margin                    |
|  +-------------------------------+   |
|  |          border               |   |
|  |  +-------------------------+  |   |
|  |  |       padding           |  |   |
|  |  |  +-------------------+  |  |   |
|  |  |  |    content        |  |  |   |
|  |  |  +-------------------+  |  |   |
|  |  +-------------------------+  |   |
|  +-------------------------------+   |
+--------------------------------------+
```

### 2.1 The Two Box-Sizing Models

| `box-sizing` | Width formula | Notes |
|:-----|:-----|:-----|
| `content-box` (default) | `outer-width = width + padding-x + border-x` | `width` = content width |
| `border-box` (modern) | `outer-width = width` | `width` includes padding + border |

Margin is **never** part of "the box" itself — it sits *outside* the border. But it does contribute to the box's *outer edge*, which is what neighbors see.

### 2.2 Content-Box — The Default

```css
.legacy {
  box-sizing: content-box; /* the spec default */
  width: 200px;
  padding: 20px;
  border: 5px solid red;
  /* outer width = 200 + 40 + 10 = 250px */
}
```

Adding padding or border *grows* the box. This is why old layouts broke when you added a 1px border: the box jumped to 251px and overflowed its column.

### 2.3 Border-Box — The Modern Reset

```css
*, *::before, *::after {
  box-sizing: border-box;
}

.modern {
  width: 200px;
  padding: 20px;
  border: 5px solid red;
  /* outer width = 200px (content shrinks to 200 - 40 - 10 = 150px) */
}
```

The universal selector applies `border-box` to every element including pseudo-elements. **This is the canonical reset.** It reverses the formula: `content-width = width - padding - border`. Adding padding or border now *shrinks* the content area without changing the outer footprint.

### 2.4 Why Border-Box Wins

When you build a 12-column grid where each column is `width: 8.333%`, adding `padding: 1rem` would, with `content-box`, push each column past the `8.333%` boundary and break wrapping. With `border-box`, the column stays exactly `8.333%` of the container, and padding eats inward.

### 2.5 Margin Is Not Part of the Box

Margins live outside the border edge. They contribute to *layout positioning* (where the next box starts) but not to the box's own width/height calculations.

```css
.box {
  width: 100px;
  margin: 20px;
  /* The box is 100px wide. Its outer edge is 140px wide (100 + 20 + 20). */
}
```

This distinction matters because:

- `getBoundingClientRect()` returns the **border-box rectangle** — margins excluded
- `offsetWidth`/`offsetHeight` are **border-box dimensions**
- A box's *containing block* uses the **content-box** of the ancestor by default

### 2.6 Background Painting Origin

The `background-clip` property controls which box edge clips the background:

```css
.card {
  background-clip: border-box;  /* default — paints under border */
  background-clip: padding-box; /* stops at inner border edge */
  background-clip: content-box; /* stops at padding edge */
}
```

This matters for translucent backgrounds with dashed borders, where the gaps in the border would reveal the background underneath.

## 3. Margin Collapse — The Algorithm

Margin collapse is the most-cursed CSS rule. It is also strictly defined in CSS 2.1 §8.3.1.

### 3.1 The Three Collapse Cases

**Case 1: Adjacent vertical margins between block-level siblings** collapse to the larger of the two.

```html
<style>
  .a { margin-bottom: 30px; }
  .b { margin-top: 20px; }
</style>
<div class="a">A</div>
<div class="b">B</div>
<!-- Gap = max(30, 20) = 30px (NOT 50px) -->
```

**Case 2: Parent and first/last child** collapse if no padding, border, or content-bearing inline content separates them.

```html
<style>
  .parent { /* no padding, no border */ }
  .child  { margin-top: 20px; }
</style>
<div class="parent">
  <div class="child">Top edge of child shows on parent's top edge</div>
</div>
<!-- The child's margin-top "leaks out" of the parent. -->
```

**Case 3: Empty block with vertical margins** collapses with itself: its `margin-top` and `margin-bottom` collapse to the larger.

```html
<style>
  .empty { margin-top: 20px; margin-bottom: 30px; }
</style>
<div class="empty"></div>
<!-- Outer footprint = 30px, not 50px. -->
```

### 3.2 The Block-Axis Restriction

Margin collapse happens **only on the block axis** (vertical in horizontal writing modes; horizontal in vertical writing modes). Horizontal margins between siblings *never* collapse — they stack.

### 3.3 Negative Margins Participate

The collapsed value with negative margins is `max(positive_margins) + min(negative_margins)`:

| Margins | Collapsed result |
|:-----|:-----|
| `30px` and `20px` | `30px` |
| `30px` and `-10px` | `30 + (-10) = 20px` |
| `-30px` and `-10px` | `min(-30, -10) = -30px` |

### 3.4 Cures — When Collapse Stops

| Cure | Mechanism |
|:-----|:-----|
| `padding` or `border` on parent | Separates parent edge from child edge → no collapse |
| `display: flex` or `display: grid` | Children participate in flex/grid layout, not BFC |
| `display: flow-root` | Establishes new BFC; isolates children's margins |
| `overflow: hidden\|auto\|scroll` | Establishes new BFC (side effect) |
| `position: absolute\|fixed` | Removed from flow; no margin participation |
| `float: left\|right` | Removed from flow; no margin participation |

### 3.5 The `display: flow-root` Cure

```css
.cure {
  display: flow-root; /* establishes a BFC; isolates child margins */
}
```

`flow-root` is the **purpose-built fix** for parent/child margin leak. Older codebases used `overflow: hidden`, which works but introduces a clipping side-effect that breaks tooltips and shadows.

## 4. Block Formatting Context (BFC) — Triggers

A Block Formatting Context is a region of the layout where boxes stack vertically and floats interact only within. The set of triggers is exact and short.

### 4.1 Formal Trigger List

A BFC is established by:

| Trigger | Property |
|:-----|:-----|
| Root element | (the `<html>` element always) |
| Floats | `float: left`, `float: right` |
| Absolute positioning | `position: absolute`, `position: fixed` |
| Inline-block | `display: inline-block` |
| Table cells | `display: table-cell`, `display: table-caption` |
| Flex containers (children of) | `display: flex`, `display: inline-flex` |
| Grid containers (children of) | `display: grid`, `display: inline-grid` |
| Flow root | `display: flow-root` |
| Overflow non-visible | `overflow: hidden\|auto\|scroll\|clip` |
| Multi-column | `column-count: <n>` or `column-width: <length>` |
| Containment | `contain: layout`, `contain: paint`, `contain: content` |

### 4.2 What a BFC Does

1. **Isolates floats**: floats inside a BFC do not extend outside; ancestor floats do not poke in.
2. **Prevents margin collapse**: child margins do not collapse with the BFC root.
3. **Contains in-flow descendants**: the BFC always extends to enclose its in-flow children, even floats.

### 4.3 The "Clear Floats" Pattern

```css
/* Anti-pattern: side-effect-y */
.parent {
  overflow: hidden; /* establishes BFC, clips children */
}

/* Modern: purpose-built */
.parent {
  display: flow-root;
}
```

`overflow: hidden` was historically used to make a parent enclose its floated children. It works (because BFC), but clips overflowing decorations. `display: flow-root` is the explicit, side-effect-free version added in CSS Display Level 3.

### 4.4 The Two-Column Float Layout (Historical)

```css
/* The old way — pre-Flexbox */
.left  { float: left;  width: 30%; }
.right { float: right; width: 70%; }
.container { display: flow-root; /* enclose floats */ }
```

Each `float` establishes its own BFC, so they sit side-by-side. The container needs `flow-root` (or any BFC trigger) to enclose them. This pattern is superseded by `display: flex` / `display: grid` in all modern code.

## 5. Inline Formatting Context (IFC) — How Lines Are Built

An Inline Formatting Context is established by a block-level container that contains inline-level children (or anonymous inline boxes wrapping text).

### 5.1 Line Boxes

The renderer walks inline content left-to-right (in LTR writing mode), packing inline boxes into a **line box**. A line box's height = the maximum of all inline-box heights it contains. When the next inline box would exceed the IFC's inline size, a new line box starts.

### 5.2 The Strut

Every line box has an invisible **strut** — a zero-width inline box with the line box's `font-family` and `font-size` and the container's `line-height`. The strut sets the minimum line-box height even if the line is empty:

```css
.empty-line {
  font-size: 16px;
  line-height: 1.5;
  /* strut height ≈ 24px, line box ≈ 24px */
}
```

### 5.3 Baseline and Vertical-Align

Inline-level boxes align relative to a baseline. The default baseline for text is the **alphabetic baseline** of the font.

```
ascender ───────────  ↑
                       │ font ascent
baseline ─────────── ─┴─
                       │ font descent
descender ───────────  ↓
```

`vertical-align` controls the alignment of inline-level boxes within their line box:

| Value | Effect |
|:-----|:-----|
| `baseline` (default) | Box's baseline aligns to line's baseline |
| `top` | Top of box at top of line box |
| `bottom` | Bottom of box at bottom of line box |
| `middle` | Box's vertical midpoint at parent's `(baseline + x-height/2)` |
| `sub` | Lowered for subscript |
| `super` | Raised for superscript |
| `text-top` | Top of box at top of parent's font |
| `text-bottom` | Bottom of box at bottom of parent's font |
| `<length>` | Specified offset above baseline |
| `<percentage>` | Percentage of the box's `line-height` |

### 5.4 White-Space Normalization

By default, inline content collapses runs of whitespace into a single space, and newlines are treated as spaces. Controlled by `white-space`:

| Value | Whitespace | Newlines | Wrap |
|:-----|:-----|:-----|:-----|
| `normal` | collapse | collapse | wrap |
| `nowrap` | collapse | collapse | no wrap |
| `pre` | preserve | preserve | no wrap |
| `pre-wrap` | preserve | preserve | wrap |
| `pre-line` | collapse | preserve | wrap |
| `break-spaces` | preserve | preserve | wrap (after spaces) |

### 5.5 `line-height` Inheritance

`line-height` is one of the few properties whose **inheritance rules differ by computed-value form**:

| `line-height` form | Inherited value |
|:-----|:-----|
| `<number>` (e.g., `1.5`) | The number itself; descendants multiply by their own `font-size` |
| `<length>` (e.g., `24px`) | The computed length; descendants inherit the same pixel value |
| `<percentage>` (e.g., `150%`) | The computed length (`font-size × percentage`); descendants inherit the resolved length |
| `normal` | `normal`; each box uses its font's metric |

This is why `body { line-height: 1.5; }` is preferred — descendants scale their leading to their own font size, not the body's.

## 6. Float — The Layout Algorithm

`float` removes a box from normal flow but maintains its space, allowing inline content to wrap around it.

### 6.1 The Float Property

```css
.fig { float: left; }
.fig { float: right; }
.fig { float: none; }  /* default */
```

A floated box is taken out of normal flow but is still part of the document. Subsequent block-level boxes flow as if the float were not there, but inline content (line boxes) flow around it.

### 6.2 The Float Algorithm

1. **Generate the float's box** with shrink-to-fit width (if width is `auto`).
2. **Position the float** at the top-most, left-most (for `float: left`) or right-most (for `float: right`) position within the containing block, without overlapping any prior floats.
3. **Subsequent line boxes shorten** to avoid the float's footprint.
4. **Subsequent block boxes** are positioned as if the float were not there, but their **line boxes** still shorten.

### 6.3 The Canonical Text-Wrap-Around

```html
<style>
  img.fig { float: left; margin: 0 1em 1em 0; }
</style>
<article>
  <img class="fig" src="diagram.png">
  <p>Lorem ipsum dolor sit amet, consectetur adipiscing elit...</p>
</article>
```

The `<p>`'s block box starts at the top of `<article>`, but its line boxes shorten to clear the floated `<img>`'s width. After the image's bottom, line boxes resume full width.

### 6.4 `clear` — Interrupting the Float

```css
.clear-both { clear: both; }   /* below all prior floats */
.clear-left { clear: left; }
.clear-right { clear: right; }
```

A box with `clear` is positioned below the lowest edge of the prior floats it clears. This is how the old "clearfix" pattern worked:

```css
/* Pre-flow-root clearfix */
.parent::after {
  content: "";
  display: block;
  clear: both;
}
```

A pseudo-element with `clear: both` forces the parent to extend below all floats. Modern code uses `display: flow-root` instead.

### 6.5 Why Float Lost Its Layout Job

Float was never designed for layout — it was designed for figures-with-text-wrap. Building two-column layouts with float required:

- Calculating width percentages by hand
- Adding `display: flow-root` (or clearfix) on the parent
- Ordering source carefully (HTML order = visual order)
- Forgetting the gutter
- Crying

Flexbox and Grid handle all of this declaratively. Float survives only for its original job: figure-with-text-wrap.

## 7. Flexbox — The Sizing Algorithm — Detailed

Flexbox is defined in CSS Flexible Box Layout Module Level 1. Section §9 (the layout algorithm) is the heart. Here is the full pass for a single flex line.

### 7.1 The Algorithm

| Step | What it computes |
|:-----|:-----|
| 1 | Available main-axis space in the container |
| 2 | Flex base size of each item |
| 3 | Hypothetical main size of each item |
| 4 | Free space (positive or negative) |
| 5 | Distribute free space via `flex-grow` / `flex-shrink` |
| 6 | Resolve cross-axis sizes via `align-items` / `align-self` |
| 7 | Apply `justify-content` (main) and `align-content` (cross) |

### 7.2 Step 1 — Available Main-Axis Space

The flex container's main-axis size minus the sum of its main-axis padding and border. This is what items must share.

### 7.3 Step 2 — Flex Base Size

Per item: resolve `flex-basis` to a length.

| `flex-basis` | Resolution |
|:-----|:-----|
| `auto` | Use the item's **main-axis content size** (a `width: 200px` becomes 200; bare content becomes max-content) |
| `<length>` | The length (e.g., `10rem` → resolved to pixels) |
| `<percentage>` | Percentage of the container's main-axis size |
| `content` | The item's content size, ignoring `width`/`height` |
| `0` | Zero — "flex from nothing" |

### 7.4 Step 3 — Hypothetical Main Size

```
hypothetical_main_size = clamp(flex_base_size, min_main_size, max_main_size)
```

The hypothetical size is the flex base clamped by `min-width`/`min-height` and `max-width`/`max-height` (mapped to the main axis).

### 7.5 Step 4 — Free Space

```
free_space = container_main_size - sum(hypothetical_main_size)
```

If positive: items want to grow into free space.
If negative: items must shrink to fit.

### 7.6 Step 5a — Distribute Positive Free Space (flex-grow)

Per item:

```
share = (item.flex_grow / sum_of_flex_grow) × free_space
item.used_main_size = item.hypothetical_main_size + share
```

**Items with `flex-grow: 0` get no share** — they stay at their hypothetical size.

### 7.7 Step 5b — Distribute Negative Free Space (flex-shrink)

Per item:

```
weighted_shrink_factor = item.flex_shrink × item.flex_base_size
share = (weighted_shrink_factor / sum_of_weighted_shrink_factors) × free_space
item.used_main_size = item.hypothetical_main_size + share   // share is negative
```

**Larger items shrink more.** This is why `flex: 1 1 0%` (basis = 0) makes items grow evenly: with basis 0, all weighted-shrink factors are zero (no shrinking happens by default, growth is even).

### 7.8 Step 6 — Cross-Axis Sizing

```
if align-self / align-items === stretch && cross-axis size === auto:
    item.cross_size = container.cross_size - item.cross_margins
else:
    item.cross_size = item's cross-axis intrinsic size or specified size
```

### 7.9 Step 7 — Distribute Items

`justify-content` distributes leftover main-axis space (after item growth):

| Value | Behavior |
|:-----|:-----|
| `flex-start` (default in row) | Pack at start |
| `flex-end` | Pack at end |
| `center` | Pack at center |
| `space-between` | Equal space between items |
| `space-around` | Equal space around each item (half-gaps at edges) |
| `space-evenly` | Equal space between *and* at edges |

`align-content` (only relevant for `flex-wrap` with multiple lines) distributes cross-axis space between lines.

### 7.10 Worked Example

```html
<style>
  .flex {
    display: flex;
    width: 600px;
  }
  .a { flex: 0 0 200px; }
  .b { flex: 1 1 0%; }
  .c { flex: 2 1 0%; }
</style>
<div class="flex">
  <div class="a">A</div>
  <div class="b">B</div>
  <div class="c">C</div>
</div>
```

- Container main-axis: 600px
- Hypothetical sizes: A=200, B=0, C=0 → sum = 200
- Free space: 600 − 200 = 400
- Sum of `flex-grow`: 0 + 1 + 2 = 3
- A gets 0 share → 200px
- B gets `(1/3) × 400 = 133.33px`
- C gets `(2/3) × 400 = 266.67px`
- Final: `200 + 133.33 + 266.67 = 600px`

## 8. Flexbox — The `min-width: 0` Story

The most-cited Flexbox bug.

### 8.1 The Default `min-width` on Flex Items

Per CSS Flexbox §4.5, the default minimum size of a flex item is `auto`, which resolves to the item's **content min-size** — the `min-content` of its contents.

Practically: a flex item containing a long word *will not shrink below the width of that word* by default. It overflows the container.

### 8.2 The Canonical Symptom

```html
<style>
  .row { display: flex; width: 300px; }
  .item {
    flex: 1;
    overflow: hidden;
    text-overflow: ellipsis;
    white-space: nowrap;
  }
</style>
<div class="row">
  <div class="item">A really really really long text that should ellipse</div>
  <div class="item">Sidebar</div>
</div>
```

**Expected:** the long text ellipses at the column edge.
**Actual:** the long text overflows because the flex item refuses to shrink below its `min-content` (the longest unbreakable word).

### 8.3 The Fix

```css
.item {
  min-width: 0; /* override the auto default */
  /* ... */
}
```

Now the item can shrink below its content's min size, and `overflow: hidden` + `text-overflow: ellipsis` work as intended.

### 8.4 Column Flex — `min-height: 0`

The same applies in column-direction flex:

```css
.col {
  display: flex;
  flex-direction: column;
  height: 100vh;
}

.col > .scroll-region {
  flex: 1;
  min-height: 0; /* required for inner overflow scrolling */
  overflow: auto;
}
```

Without `min-height: 0`, the scroll region's `min-content` height (the height of all its children) prevents shrinking, so `overflow: auto` never engages.

### 8.5 The "Spec Default Trap"

This trap exists because `min-width: auto` (computed) was added to Flexbox to prevent unintentional truncation. The spec authors chose safety over composability; the result is that every serious flex layout sets `min-width: 0` on items that contain text.

## 9. Flexbox — `flex` Shorthand Decoded

The `flex` shorthand sets `flex-grow`, `flex-shrink`, and `flex-basis` together. Its keyword forms are not interchangeable.

### 9.1 The Forms

| `flex` | grow | shrink | basis | Meaning |
|:-----|:-----|:-----|:-----|:-----|
| `0` | 0 | 1 | 0% | Don't grow, can shrink, basis = 0 (rarely useful) |
| `1` | 1 | 1 | 0% | Equal-share grow, can shrink, basis = 0 |
| `2` | 2 | 1 | 0% | 2× share grow |
| `auto` | 1 | 1 | auto | Grow + shrink from content size |
| `initial` | 0 | 1 | auto | Don't grow, can shrink, basis = content (default!) |
| `none` | 0 | 0 | auto | Inflexible — won't grow or shrink |
| `1 1 auto` | 1 | 1 | auto | Equal-share grow, basis = content |
| `1 1 0%` | 1 | 1 | 0% | Equal-share grow, basis = 0 |

### 9.2 The Most Common Confusion: `flex: 1` vs `flex: 1 1 auto`

```css
.equal-cols .item { flex: 1; }       /* basis: 0; truly equal columns */
.flexible    .item { flex: 1 1 auto; } /* basis: content; bigger items get more */
```

With `flex: 1` (basis = 0), each item starts from zero and grows proportionally to `flex-grow`. Three items with `flex: 1` are exactly equal width.

With `flex: 1 1 auto` (basis = content), each item starts at its content size and *then* shares free space. An item with longer content is larger.

### 9.3 The Default — `flex: initial`

```css
/* CSS default for any flex item that doesn't set `flex` */
.item {
  flex: 0 1 auto;
  /* don't grow; can shrink; basis = content */
}
```

This is why a row of flex items without `flex: 1` collapses to content sizes when the container has extra space.

### 9.4 The Inflexible — `flex: none`

```css
.fixed-sidebar {
  flex: none; /* equivalent to: 0 0 auto */
  width: 240px;
}
```

`flex: none` forbids the item from growing or shrinking — its size is its specified `width`/`height` or content size.

### 9.5 Choosing the Right Form

| Goal | Use |
|:-----|:-----|
| Equal-width columns | `flex: 1` (basis 0) |
| Content-sized + share extra | `flex: 1 1 auto` (basis content) |
| Fixed-width sidebar | `flex: none; width: 240px` |
| Don't grow but allow shrink | `flex: 0 1 auto` (default) |
| Grow only when room available | `flex: 1 0 auto` (basis content, can grow, no shrink) |

## 10. Grid — The Two-Step Algorithm

CSS Grid Layout (Module Level 1, W3C Recommendation 2020) defines the most powerful 2D layout algorithm in CSS. The high-level pass has four conceptual phases.

### 10.1 Phase 1 — Resolve the Grid Template

The container's `grid-template-rows`, `grid-template-columns`, and `grid-template-areas` are parsed into a **definite track list** (lengths, percentages, `fr` units, `min-content`, `max-content`, `minmax()`, `auto`, `repeat()`, `fit-content()`).

```css
.grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr) 200px;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "header header header header"
    "main   main   main   side  "
    "footer footer footer footer";
}
```

The renderer expands `repeat(3, 1fr)` to three tracks, then resolves all line numbers and named lines.

### 10.2 Phase 2 — Place Items

1. **Explicit placement first** — items with `grid-area`, `grid-column`, or `grid-row` are placed at their named positions or numeric line spans.
2. **Auto-placement** — remaining items walk the grid following `grid-auto-flow`:
   - `row` (default) — left-to-right, top-to-bottom
   - `column` — top-to-bottom, left-to-right
   - `dense` — back-fill empty cells with later items that fit

Items placed beyond the explicit grid create **implicit tracks** sized by `grid-auto-rows` / `grid-auto-columns`.

### 10.3 Phase 3 — Resolve Track Sizes

The track-sizing algorithm (CSS Grid §11) runs on rows and columns separately:

1. Initialize each track's base size and growth limit from its definition.
2. Resolve intrinsic minimums by scanning items' min-content/max-content.
3. Distribute available space first to satisfy minimums.
4. Distribute leftover space to maximums.
5. Distribute remaining space across `fr` tracks proportional to their flex factors.
6. Apply `minmax()` clamps.

### 10.4 Phase 4 — Position Items

Each item is placed in its grid area. Within the area, `justify-self` (inline-axis) and `align-self` (block-axis) control alignment:

| Value | Behavior |
|:-----|:-----|
| `stretch` (default) | Fill the area |
| `start` | Pack at start |
| `end` | Pack at end |
| `center` | Pack at center |
| `baseline` | Align baselines |

`justify-items` / `align-items` set the default for all items in the container. `justify-content` / `align-content` distribute extra space when the grid is smaller than the container.

### 10.5 Auto-Flow Dense Packing

```css
.gallery {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  grid-auto-flow: row dense;
}
.gallery .wide { grid-column: span 2; }
```

`dense` makes auto-placement back-fill earlier holes when a later item fits. This builds compact masonry-like layouts but reorders visual flow from source order — be deliberate about accessibility implications.

### 10.6 Subgrid (Level 2)

Subgrid lets a nested grid inherit its parent's tracks rather than defining its own. Detailed in §12.

## 11. Grid — The Track Sizing Algorithm

The track-sizing algorithm is defined in CSS Grid Layout Level 1 §11. It is a 12-step procedure that runs once per axis per layout pass.

### 11.1 The Twelve Steps (§11.5)

| Step | Phase | What it does |
|:-----|:-----|:-----|
| 1 | Initialize | Set each track's `base size` and `growth limit` from its definition |
| 2 | Resolve intrinsic sizes | Walk items; compute min-content/max-content contributions |
| 3 |  | Increase base sizes to satisfy intrinsic minimums |
| 4 |  | Increase growth limits if min > max |
| 5 | Maximize tracks | Distribute remaining free space up to growth limits |
| 6 | Expand flex tracks | Find the flex fraction; size `fr` tracks proportionally |
| 7 |  | Clamp by `max-track-sizing-function` |
| 8 | Stretch auto tracks | If `align-content: stretch`, expand `auto` tracks to fill |
| 9 | Item placement | Position items in their grid areas |
| 10 | Spanning items | Resolve sizes for items spanning multiple tracks |
| 11 | Final pass | Apply `align-self` / `justify-self` |
| 12 | Layout overflow | Compute overflow if items exceed track sizes |

### 11.2 The `fr` Unit

The `fr` (fraction) unit represents a share of leftover free space:

```css
.three-col {
  display: grid;
  grid-template-columns: 1fr 2fr 1fr;
  /* leftover space split 1:2:1 → 25% / 50% / 25% */
}
```

`fr` tracks are sized **after** all fixed and intrinsic tracks. The flex fraction is computed as:

```
flex_fraction = leftover_space / sum_of_flex_factors
```

If `min-content` of an `fr` track exceeds its calculated share, it grows to fit (preventing overflow).

### 11.3 `minmax()` Resolution

```css
.responsive {
  grid-template-columns: minmax(200px, 1fr);
  /* track is at least 200px, at most 1fr of free space */
}
```

`minmax(min, max)` provides a lower and upper bound. If `max < min`, `max` is treated as `min`.

### 11.4 `auto-fit` vs `auto-fill`

Both are arguments to `repeat()`:

```css
.gallery {
  grid-template-columns: repeat(auto-fit,  minmax(200px, 1fr));
  grid-template-columns: repeat(auto-fill, minmax(200px, 1fr));
}
```

| Keyword | Empty tracks |
|:-----|:-----|
| `auto-fit` | Collapses empty tracks to 0; remaining tracks grow to fill |
| `auto-fill` | Keeps empty tracks at their min size |

Visually with 600px container, 200px min, 1fr max, and 2 items:

- `auto-fit`: 2 columns × 300px (items grow to fill)
- `auto-fill`: 3 columns × 200px (one empty), with 0px leftover

`auto-fit` is more common for galleries; `auto-fill` is useful when you want consistent column widths regardless of item count.

### 11.5 Cost Model

Track sizing is `O(items × tracks)` per layout pass. Modern browsers (Blink, WebKit, Gecko) optimize the inner loop, but very large grids (10,000+ items) can show measurable time. Use `contain: layout` on offscreen sub-grids to scope cost.

## 12. Subgrid (CSS Grid Level 2)

Subgrid is the answer to "I want this nested grid to share its parent's tracks." Defined in CSS Grid Level 2.

### 12.1 The Problem

```html
<style>
  .parent { display: grid; grid-template-columns: 1fr 1fr 1fr; }
  .card   { display: grid; grid-template-rows: auto 1fr auto; }
</style>
<div class="parent">
  <article class="card"><h3>Title</h3><p>Body</p><button>CTA</button></article>
  <article class="card">...</article>
  <article class="card">...</article>
</div>
```

Each `.card` has its own row tracks. Different content lengths mean buttons end up at different vertical positions across cards. The fix used to be table-like alignment hacks.

### 12.2 The Subgrid Solution

```css
.parent {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  grid-template-rows: repeat(3, auto);
}

.card {
  display: grid;
  grid-row: span 3;
  grid-template-rows: subgrid; /* inherit parent's row tracks */
}
```

Now all cards align their rows (title, body, CTA) to the parent's tracks. Headers across cards align horizontally; buttons align horizontally; bodies stretch independently.

### 12.3 Per-Axis Subgrid

Subgrid can apply to one axis or both:

```css
.col-only { grid-template-columns: subgrid; grid-template-rows: 200px 1fr; }
.row-only { grid-template-columns: 1fr 1fr; grid-template-rows: subgrid; }
.both     { grid-template-columns: subgrid; grid-template-rows: subgrid; }
```

### 12.4 Inherited Line Names

A subgrid inherits its parent's line names within its span:

```css
.parent {
  grid-template-columns: [start] 1fr [main] 2fr [end];
}
.subgrid {
  grid-template-columns: subgrid;
  /* can use: grid-column: start / main; */
}
```

This is invaluable for design systems: define line names once at the page level, reference them from cards.

### 12.5 The Canonical Card-Grid Use Case

Cards with shared horizontal rhythm (headers aligned, footers aligned, body grows independently) is the case `subgrid` was designed for. Before subgrid, this required either fixed heights, JavaScript measurement, or tortuous `display: grid` plus `flex` combinations.

## 13. Positioning — Absolute / Fixed / Sticky

The `position` property selects the layout scheme for an element. Defined in CSS Positioned Layout Level 3.

### 13.1 The Five Values

| `position` | Containing block | In flow? | Offset properties (`top` etc.) |
|:-----|:-----|:-----|:-----|
| `static` (default) | n/a | yes | ignored |
| `relative` | (no change) | yes | offset from normal-flow position |
| `absolute` | nearest **positioned** ancestor | no | offset from containing block |
| `fixed` | viewport (or transform-having ancestor) | no | offset from containing block |
| `sticky` | scroll container | yes (initially) | offset from container until threshold |

### 13.2 `position: relative` — Anchor Without Removing

```css
.anchor {
  position: relative;
  /* No visual change unless top/left/etc set */
  /* Establishes a containing block for absolutely-positioned descendants */
}
```

Two effects:
1. The element offsets from its normal-flow position by `top`/`right`/`bottom`/`left` *without* affecting siblings (the in-flow space is reserved at the un-offset position).
2. The element becomes the **containing block** for any `position: absolute` descendant.

### 13.3 `position: absolute` — Out of Flow

```css
.tooltip {
  position: absolute;
  top: 100%;
  left: 0;
  /* Positioned relative to nearest positioned ancestor */
}
```

The element is taken out of normal flow; subsequent elements lay out as if it were not there. Its containing block is the **nearest ancestor with `position` other than `static`** — or the **initial containing block** (the viewport at full-document size) if none.

### 13.4 `position: fixed` — Relative to Viewport

```css
.modal-backdrop {
  position: fixed;
  inset: 0; /* top: 0; right: 0; bottom: 0; left: 0; */
  /* Covers the viewport regardless of scroll */
}
```

Fixed elements are positioned relative to the viewport (initial containing block) and do not move with scroll. **Exception**: a fixed element nested inside an ancestor with `transform`, `perspective`, `filter`, `backdrop-filter`, `contain: paint`, or `will-change: transform` becomes positioned relative to **that ancestor**, not the viewport. (See §14.)

### 13.5 `position: sticky` — Hybrid

```css
.toc {
  position: sticky;
  top: 16px;
  /* Behaves as `relative` until scroll-threshold; then `fixed` to that threshold within the scroll container */
}
```

Sticky elements have four conditions for sticking:

1. The element must have at least one of `top`, `right`, `bottom`, `left` set.
2. The element must be inside a **scroll container** (an ancestor with `overflow: scroll\|auto\|hidden`).
3. The scroll container must have **enough room** to scroll past the element's "stick threshold."
4. No ancestor between the sticky element and the scroll container can have `overflow: hidden` (it would clip the sticky behavior).

### 13.6 The "Sticky Doesn't Work" Checklist

1. Did you set `top`/`bottom`?
2. Is there a scroll container ancestor?
3. Is the parent **tall enough** to scroll? (Sticky cannot exit its parent.)
4. Does any ancestor have `overflow: hidden\|clip` between the element and the scroll container?
5. For tables: `position: sticky` on `<th>` requires `border-collapse: separate` (Chrome historical bug — fixed in modern browsers but worth checking).

## 14. Containing Block Determination

The **containing block** is the rectangle inside which a box is positioned and sized. Its rules are deceptively subtle.

### 14.1 Formal Rules (CSS 2.1 §10.1)

| Element's `position` | Containing block |
|:-----|:-----|
| `static` or `relative` | Content edge of nearest **block-level** ancestor |
| `absolute` | Padding edge of nearest **positioned** ancestor (else initial containing block) |
| `fixed` | Initial containing block (viewport) **or** padding edge of nearest ancestor with `transform`, `perspective`, `filter`, `backdrop-filter`, `will-change: transform`, `contain: paint`/`layout`, etc. |
| Root element | Initial containing block (viewport-sized) |

### 14.2 The Transform Surprise

```html
<style>
  .carousel { transform: translateZ(0); /* GPU promotion */ }
  .modal    { position: fixed; inset: 0; }
</style>
<div class="carousel">
  <div class="modal">Why doesn't this cover the viewport?</div>
</div>
```

Because `.carousel` has a non-`none` `transform`, it becomes the containing block for `position: fixed` descendants. The modal covers `.carousel`, not the viewport.

### 14.3 Why Transform Changes the Containing Block

The CSS Transforms Module §6.3 specifies that a transformed element creates a containing block for all descendants — even fixed ones — because the transform moves the element's coordinate system. Fixed positioning relative to the viewport would break inside a transformed parent.

The same applies to:
- `transform: <anything-other-than-none>`
- `perspective: <anything-other-than-none>`
- `filter: <anything-other-than-none>`
- `backdrop-filter: <anything-other-than-none>`
- `will-change: transform` (in some browsers)
- `contain: paint`
- `contain: layout`
- `contain: strict`

### 14.4 Debugging Containing Blocks

In Chrome DevTools: select an element, inspect Computed → Position → "Containing Block." The pane shows the actual containing block, including any transformed ancestor that captured fixed positioning.

## 15. Stacking Contexts

Stacking contexts control which boxes paint on top of which. Z-index is **only meaningful within a stacking context**, not globally.

### 15.1 Stacking-Context Triggers

A stacking context is established by:

| Trigger | Property |
|:-----|:-----|
| Root element | (always) |
| Positioned + z-index | `position: absolute\|relative\|fixed\|sticky` AND `z-index: <number>` |
| Position fixed/sticky | `position: fixed`, `position: sticky` (always create stacking context regardless of z-index) |
| Opacity < 1 | `opacity: <less-than-1>` |
| Transform | `transform: <not-none>` |
| Filter | `filter: <not-none>` |
| Backdrop-filter | `backdrop-filter: <not-none>` |
| Mix-blend-mode | `mix-blend-mode: <not-normal>` |
| Isolation | `isolation: isolate` |
| Will-change | `will-change: transform\|opacity\|filter` |
| Containment | `contain: layout`, `contain: paint`, `contain: strict`, `contain: content` |
| Mask | `mask: <not-none>`, `clip-path: <not-none>` |
| Flex/Grid item with z-index | A child of `display: flex\|grid` with `z-index: <number>` |

### 15.2 The Painting Order Within a Stacking Context

Per CSS 2.1 Appendix E (and the Position 3 extension):

1. Background and borders of the stacking-context root
2. Descendants with negative `z-index` (stacking-context order)
3. In-flow non-positioned descendants (block-level)
4. Non-positioned floats
5. In-flow non-positioned descendants (inline-level)
6. Positioned descendants with `z-index: auto` or `z-index: 0` (in DOM order)
7. Positioned descendants with positive `z-index` (stacking-context order)

### 15.3 Z-Index Is Local

```html
<style>
  .a { position: relative; z-index: 1;     opacity: 0.99; }
  .b { position: relative; z-index: 99999; }
</style>
<div class="a">
  <div class="b">I will not appear above the .a sibling</div>
</div>
<div class="a">other sibling</div>
```

`.a`'s `opacity: 0.99` creates a stacking context. `.b`'s z-index of 99999 is local to `.a`'s stacking context — it cannot escape to compete with the other `.a` sibling.

### 15.4 The `isolation: isolate` Trick

Embedded third-party widgets (chat, payments, ads) sometimes use absurd z-index values like `2147483647`. They can paint over your modals. The fix:

```css
.app-shell {
  isolation: isolate; /* establishes a stacking context */
}

.modal-portal {
  position: fixed;
  z-index: 1000; /* now competes only within app-shell */
}
```

`isolation: isolate` creates a stacking context without any visual side-effect — the only purpose-built way to corral z-index battles.

### 15.5 Common Stacking-Context Bugs

- **`opacity: 0.99` for performance hints** — accidentally creates a stacking context, breaking sibling z-index assumptions.
- **`will-change: transform` for promotion** — same issue.
- **`transform: translateZ(0)` GPU hack** — same issue.
- **`position: fixed` with no z-index** — establishes a stacking context AND becomes the containing block for descendants.

The safe rule: **don't use z-index without understanding which stacking context you are in**.

## 16. Container Queries — The Containment Requirement

Container queries (CSS Containment Module Level 3) let an element respond to its **parent container's size**, not the viewport.

### 16.1 The Setup

```css
.card-host {
  container-type: inline-size; /* the parent declares it queryable */
  container-name: card;        /* optional name */
}

@container card (min-width: 400px) {
  .card { display: grid; grid-template-columns: 200px 1fr; }
}

@container card (max-width: 399px) {
  .card { display: block; }
}
```

The browser observes the `.card-host`'s inline size. When it crosses 400px, the contained `.card` elements switch layouts.

### 16.2 `container-type` Values

| Value | What's queryable |
|:-----|:-----|
| `normal` (default) | Nothing — element is not a query container |
| `inline-size` | Inline-axis size only |
| `size` | Both axes (more expensive) |

`size` requires the container to have a definite block size, which usually means setting `height` explicitly. `inline-size` works without that constraint and is the common choice.

### 16.3 Container Units

Container queries enable a new family of length units:

| Unit | Meaning |
|:-----|:-----|
| `cqw` | 1% of the query container's width |
| `cqh` | 1% of the query container's height |
| `cqi` | 1% of the inline size |
| `cqb` | 1% of the block size |
| `cqmin` | smaller of `cqi` / `cqb` |
| `cqmax` | larger of `cqi` / `cqb` |

```css
.card h3 {
  font-size: clamp(1rem, 4cqi, 1.5rem);
  /* Headline scales with parent card width, not viewport */
}
```

### 16.4 Style Queries (Level 5, 2023+)

```css
@container style(--theme: dark) {
  .card { background: #111; color: #eee; }
}
```

Style queries match on custom-property values rather than size. Useful for theming subtrees without class proliferation.

### 16.5 The Win

Container queries finally let components be **portable** across layouts. A card that looks compact in a sidebar and roomy in a hero panel needs only one CSS rule, not two layout-specific class variants.

## 17. Aspect-Ratio

The `aspect-ratio` property gives any element an intrinsic aspect ratio.

### 17.1 The Property

```css
.video {
  width: 100%;
  aspect-ratio: 16 / 9;
  /* height = width × (9/16) = 56.25% of width */
}

.square { aspect-ratio: 1; }   /* 1 / 1 */
.golden { aspect-ratio: 1.618; }
```

When `aspect-ratio` is set and one axis is `auto`, the other axis is computed from the ratio.

### 17.2 The Old Way — Padding-Bottom Hack

```css
/* Pre-aspect-ratio */
.video-wrap {
  position: relative;
  padding-bottom: 56.25%; /* 9 / 16 = 0.5625 */
  height: 0;
}
.video-wrap iframe {
  position: absolute;
  inset: 0;
  width: 100%;
  height: 100%;
}
```

This worked because `padding-*: <percentage>` resolves against the parent's **inline size** (width). Setting `padding-bottom: 56.25%` made the box's height a fixed fraction of its width. But it required absolutely positioning the content inside.

### 17.3 With `aspect-ratio` — Direct

```css
.video-wrap {
  width: 100%;
  aspect-ratio: 16 / 9;
}
.video-wrap iframe {
  width: 100%;
  height: 100%;
}
```

No positioning hack, no padding trick, works on any element including images, videos, and grid items.

### 17.4 Implicit Aspect Ratio for `<img>`/`<video>`

Modern browsers automatically apply `aspect-ratio` based on `width` and `height` attributes:

```html
<img src="hero.jpg" width="1600" height="900">
```

The browser computes `aspect-ratio: 1600 / 900` and reserves the layout slot before the image loads — preventing the cumulative-layout-shift jank that plagued the web from 1995–2020.

## 18. Layout Performance

Layout is the most expensive phase of the rendering pipeline for many applications.

### 18.1 What Triggers Layout

| Operation | Triggers layout? |
|:-----|:-----|
| Mutate `style.width`, `style.height`, `style.padding`, etc. | Yes |
| Mutate `style.display` | Yes |
| Insert/remove DOM nodes | Yes |
| Read `offsetWidth`, `offsetHeight`, `offsetTop`, `offsetLeft` | **Forces synchronous layout** |
| Read `getBoundingClientRect()` | **Forces synchronous layout** |
| Read `getComputedStyle()` (for layout-affected props) | **Forces synchronous layout** |
| Read `scrollTop`, `scrollLeft` (sometimes) | **Forces synchronous layout** |
| Mutate `transform`, `opacity` (when isolated) | No (composite-only) |

### 18.2 Layout Thrashing

The classic anti-pattern:

```javascript
// WRONG — interleaved reads and writes
for (const el of items) {
  el.style.width = el.offsetWidth + 10 + 'px';
  // each iteration: write-then-read forces synchronous layout
}
```

Each write invalidates layout. The next read forces a synchronous re-layout to produce a fresh `offsetWidth`. With 1000 items, you trigger 1000 synchronous layouts. Frame budget gone.

### 18.3 The Fix: Batch Reads, Then Batch Writes

```javascript
// RIGHT — read all first, then write all
const widths = items.map(el => el.offsetWidth); // one layout
items.forEach((el, i) => {
  el.style.width = (widths[i] + 10) + 'px';     // batched writes
});
```

One layout pass at the start; writes coalesce into one final layout at the next frame.

### 18.4 `requestAnimationFrame` for Reads

```javascript
function measure() {
  requestAnimationFrame(() => {
    const rect = element.getBoundingClientRect();
    // do something with rect
  });
}
```

`requestAnimationFrame` runs before the next paint, after layout has settled. Reading inside an rAF callback minimizes the chance of forcing a synchronous extra layout.

### 18.5 Layout Containment

```css
.card {
  contain: layout;
  /* Layout cost of children does not propagate to ancestors */
}

.card {
  contain: layout paint;
  /* Plus: children cannot paint outside the box */
}

.card {
  contain: strict; /* layout + paint + size */
}
```

`contain: layout` tells the browser that this element's layout is independent — changing a child's size won't relayout the rest of the page. Useful for cards, list items, and any repeated component.

### 18.6 Cost Hierarchy

| Operation | Typical cost (per frame, 1000 elements) |
|:-----|:-----|
| Composite-only (transform, opacity) | < 1 ms |
| Paint (color, background) | 1–5 ms |
| Layout (size, position) | 5–50 ms |
| Layout + nested grid resolution | 10–100 ms |

A 60fps frame budget is 16.67ms. A single bad layout pass eats the budget.

## 19. The Animation Frame Pipeline

The browser's render pipeline has four stages. Knowing which stage a property triggers is the difference between 60fps and 6fps.

### 19.1 The Four Stages

```
Style    → Layout    → Paint     → Composite
(cascade)  (boxes)     (pixels)    (GPU layers)
```

### 19.2 Property Cost Table

| Property | Triggers |
|:-----|:-----|
| `width`, `height`, `top`, `left`, `padding`, `margin`, `display` | Style + Layout + Paint + Composite |
| `color`, `background`, `border-color`, `box-shadow` | Style + Paint + Composite |
| `transform`, `opacity`, `filter` (if isolated to layer) | Style + Composite |

### 19.3 The "Animate Transform, Not Left" Rule

```css
/* SLOW — triggers layout every frame */
@keyframes slide-bad {
  from { left: 0; }
  to   { left: 100px; }
}

/* FAST — composite-only */
@keyframes slide-good {
  from { transform: translateX(0); }
  to   { transform: translateX(100px); }
}
```

`left` is a layout-axis property — animating it re-runs the entire layout pipeline 60 times a second. `transform` is a composite-only property — the GPU shifts the existing layer's coordinates without recomputing pixels.

### 19.4 `will-change` — Preemptive Promotion

```css
.draggable {
  will-change: transform;
  /* Tells the browser: I will animate transform soon. Promote me to a composite layer now. */
}
```

`will-change` reserves a GPU layer ahead of time, avoiding the one-frame jank of layer creation when the animation starts. Use sparingly — every layer costs GPU memory.

### 19.5 GPU Layer Triggers

| Trigger | Layer? |
|:-----|:-----|
| `transform: translateZ(0)` | Yes (legacy GPU hack) |
| `transform: translate3d(0,0,0)` | Yes |
| `will-change: transform` | Yes |
| `will-change: opacity` | Sometimes (browser heuristic) |
| `position: fixed` | Sometimes |
| `<video>`, `<canvas>` | Yes |
| `iframe` | Yes |
| `filter`, `backdrop-filter` | Yes |

Too many layers exhaust GPU memory; too few make animations stutter. Use Chrome DevTools → Layers panel to audit.

### 19.6 Compositor Threads

Modern browsers run the compositor on a separate thread from the main thread. Composite-only animations (transform, opacity) run on the compositor thread and continue smoothly even if the main thread is jammed by JavaScript. Layout-triggering animations block on the main thread and stutter when JavaScript runs.

## 20. Idioms at the Internals Depth

The patterns that show up in every modern codebase, with notes on the algorithms they exercise.

### 20.1 The Modern Reset

```css
*, *::before, *::after {
  box-sizing: border-box;
}

html { line-height: 1.5; }

body {
  margin: 0;
  font-family: system-ui, sans-serif;
  text-rendering: optimizeLegibility;
}

img, picture, video, canvas, svg {
  display: block;
  max-width: 100%;
}

button, input, select, textarea {
  font: inherit;
}
```

Touches: §2 (border-box), §5 (line-height inheritance number form), §17 (img max-width prevents overflow).

### 20.2 Centering — Seven Ways

| Method | CSS | Notes |
|:-----|:-----|:-----|
| Flex | `display: flex; align-items: center; justify-content: center;` | Easiest; one BFC |
| Grid + place-items | `display: grid; place-items: center;` | One-line |
| Absolute + transform | `position: absolute; top: 50%; left: 50%; transform: translate(-50%, -50%);` | Works for unknown size; creates stacking context |
| Margin auto | `margin: auto;` (in flex/grid item) | Both axes, in flex/grid |
| Text-align + line-height | `text-align: center; line-height: <h>;` | Inline content only |
| Table-cell | `display: table-cell; vertical-align: middle;` | Legacy |
| `inset: 0; margin: auto;` | Sets all four offsets to 0 + margin auto | Centers absolutely positioned box of known size |

### 20.3 Responsive Card Grid

```css
.cards {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(min(100%, 280px), 1fr));
  gap: 1rem;
}
```

Touches: §11 (auto-fit, minmax, fr resolution). The `min(100%, 280px)` inner clamp prevents single-column overflow on narrow viewports.

### 20.4 Holy Grail Layout

```css
.layout {
  display: grid;
  min-height: 100vh;
  grid-template-rows: auto 1fr auto;
  grid-template-columns: 200px 1fr 200px;
  grid-template-areas:
    "header header header"
    "left   main   right "
    "footer footer footer";
}

header { grid-area: header; }
```

### 20.9 Sticky Sidebar

```css
.layout {
  display: grid;
  grid-template-columns: 240px 1fr;
  align-items: start; /* CRITICAL for sticky to work */
}

.sidebar {
  position: sticky;
  top: 1rem;
  align-self: start;
}
```

`align-items: start` (or `align-self: start` on the item) is required because the default `stretch` would stretch the sidebar to the full grid row height, leaving no room to "scroll past" the sticky threshold.

### 20.14 Avoiding Layout Thrashing

```javascript
function measureAndApply() {
  // 1. Read all measurements
  const measurements = items.map(el => el.getBoundingClientRect());

  // 2. Compute (no DOM access)
  const updates = measurements.map(rect => ({
    width: rect.width + 10,
    height: rect.height,
  }));

  // 3. Write all
  items.forEach((el, i) => {
    el.style.width = updates[i].width + 'px';
  });
}

requestAnimationFrame(measureAndApply);
```

Touches: §18 (batch reads then writes).

## 22. Complexity

| Operation | Cost |
|:-----|:-----|
| Style resolution (cascade) | O(rules × elements), heuristically much less in practice |
| Single Block Formatting Context layout | O(children) — sequential block stack |
| Inline Formatting Context line breaking | O(inline boxes × line attempts) — usually one pass |
| Flex layout (single line) | O(items) — distribute free space in one pass |
| Flex layout (multi-line wrap) | O(items × lines) — line breaking + per-line distribution |
| Grid track sizing | O(items × tracks) — 12-step algorithm of CSS Grid §11 |
| Subgrid resolution | O(items × tracks) per nested level |
| Sticky positioning scroll handler | O(sticky elements) per scroll event |
| Layout thrashing (read-write loop) | O(N²) — each iteration triggers a fresh layout pass |
| Composite-only animation | O(layer count) per frame — runs on compositor thread |
| Layout-triggering animation | O(affected subtree) per frame — runs on main thread |
| Container query evaluation | O(query containers × queries) per resize |

Frame budget at 60fps is 16.67ms; at 120fps is 8.33ms. Layout that exceeds 5–8ms in a single subtree begins to risk frame drops once paint and composite are added. `contain: layout` is the primary tool for scoping layout cost to a subtree.
