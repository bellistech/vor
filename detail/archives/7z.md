# The Mathematics of 7z — Solid Archive and LZMA Internals

> *7z is a high-compression archive format supporting solid compression, multiple algorithms, and encryption. The math covers solid block efficiency, BCJ filters, multi-method pipelines, and AES-256 encryption overhead.*

---

## 1. Solid Compression — Cross-File Redundancy

### The Model

Solid compression treats multiple files as a single continuous stream. This exploits redundancy across file boundaries.

### Solid vs Non-Solid

$$\text{Non-Solid:} \quad \text{Size} = \sum_{i=1}^{n} C(f_i) \quad (\text{each file compressed independently})$$

$$\text{Solid:} \quad \text{Size} = C(f_1 \| f_2 \| \ldots \| f_n) \quad (\text{concatenated, then compressed})$$

$$\text{Solid Improvement} = 1 - \frac{C(\text{solid})}{C(\text{non-solid})}$$

### Worked Examples

| Content | Non-Solid | Solid | Improvement |
|:---|:---:|:---:|:---:|
| 1000 similar HTML files | 50 MiB | 5 MiB | 90% |
| Source code (many .c, .h) | 30 MiB | 8 MiB | 73% |
| Mixed file types | 200 MiB | 180 MiB | 10% |
| Already compressed (JPEGs) | 500 MiB | 498 MiB | 0.4% |

### Solid Block Size

$$\text{Solid Block Size} = \text{Number of files per block} \times \text{Avg file size}$$

| Solid Block | Files per Block | Random Access | Compression |
|:---:|:---:|:---:|:---:|
| 1 file (non-solid) | 1 | O(1) | Worst |
| 1 MiB | ~10-100 | O(1M) | Good |
| 16 MiB | ~100-1000 | O(16M) | Better |
| Unlimited | All | O(archive) | Best |

**Trade-off:** Solid compression destroys random access — extracting one file requires decompressing the entire solid block.

## 2. Compression Pipeline — Filter Chains

### The Model

7z applies a **chain of filters** before the final compressor:

$$\text{Output} = \text{Compressor}(\text{Filter}_n(\ldots \text{Filter}_1(\text{Input})))$$

### Common Filter Chains

| Input Type | Chain | Effect |
|:---|:---|:---|
| x86 executables | BCJ + LZMA2 | +5-10% better ratio |
| ARM binaries | BCJ-ARM + LZMA2 | +5-10% better ratio |
| PowerPC binaries | BCJ-PPC + LZMA2 | +5-10% better ratio |
| General data | LZMA2 only | Standard compression |
| Text | Delta + LZMA2 | Minor improvement |

### BCJ (Branch/Call/Jump) Filter

BCJ converts relative addresses in executable code to absolute addresses, making them more compressible:

$$\text{Before BCJ:} \quad \texttt{E8 xx xx xx xx} \text{ (relative call)}$$

$$\text{After BCJ:} \quad \texttt{E8 AA AA AA AA} \text{ (absolute, more redundant)}$$

### Improvement from BCJ

| Binary Type | Without BCJ | With BCJ | Improvement |
|:---|:---:|:---:|:---:|
| x86 .exe | 3.5x | 4.2x | +20% |
| x86 .dll | 3.2x | 3.8x | +19% |
| ARM .so | 3.0x | 3.5x | +17% |
| Non-executable | 5.0x | 5.0x | 0% |

## 3. LZMA2 Parameters in 7z

### Dictionary Size Impact

$$\text{7z Dictionary} = 2^{n} \text{ bytes, } n = 12 \text{ to } 30$$

| Level | Dictionary | Compress RAM | Ratio (source code) |
|:---:|:---:|:---:|:---:|
| -mx1 | 64 KiB | 2 MiB | 3x |
| -mx3 | 4 MiB | 32 MiB | 5x |
| -mx5 | 16 MiB | 192 MiB | 6.5x |
| -mx7 | 32 MiB | 384 MiB | 7x |
| -mx9 | 64 MiB | 800 MiB | 7.5x |
| -mx9 -md=1g | 1 GiB | 12 GiB | 8.5x |

### Word Size (Fast Bytes)

$$\text{Fast Bytes} = \text{nice\_len} = 8 \text{ (fast)} \rightarrow 273 \text{ (slow, best)}$$

Higher fast bytes = longer matches found = better compression = slower.

## 4. AES-256 Encryption

### The Model

7z uses AES-256-CBC with key derivation from PBKDF2-SHA256.

### Key Derivation

$$\text{Key} = \text{PBKDF2}(\text{password}, \text{salt}, 2^{19} \text{ iterations})$$

$$T_{key\_derivation} = 2^{19} \times T_{SHA256} \approx 0.5 \text{ sec on modern CPU}$$

### Encryption Overhead

$$\text{Encrypted Size} = \text{Compressed Size} + \text{IV (16 bytes)} + \text{Padding (0-15 bytes)}$$

$$\text{AES Throughput} \approx 5-10 \text{ GiB/s with AES-NI}$$

Encryption overhead is negligible compared to compression time.

### Security Comparison

| Feature | ZIP (ZipCrypto) | ZIP (AES-256) | 7z (AES-256) |
|:---|:---|:---|:---|
| Algorithm | Stream cipher | AES-256-CTR | AES-256-CBC |
| Key derivation | CRC-based | PBKDF2 | PBKDF2 |
| Iterations | N/A | 1,000 | 524,288 |
| Header encryption | No | No | Yes (optional) |
| Known attacks | Broken | Secure | Secure |

**7z encrypts filenames** when header encryption is enabled; ZIP cannot.

## 5. Multi-Volume Splitting

### The Model

$$\text{Volumes} = \lceil \frac{\text{Archive Size}}{\text{Volume Size}} \rceil$$

### Common Volume Sizes

| Target | Volume Size | Use Case |
|:---|:---:|:---|
| Email attachment | 25 MiB | Email limits |
| FAT32 filesystem | 4 GiB | USB drives |
| DVD | 4.7 GiB | Optical media |
| Blu-ray | 25 GiB | Optical media |

## 6. Format Comparison — 7z vs Alternatives

### Compression Ratio (Mixed Source Code, 1 GiB)

| Format | Size | Ratio | Time | Random Access |
|:---|:---:|:---:|:---:|:---:|
| tar.gz -9 | 180 MiB | 5.7x | 100s | No |
| tar.xz -6 | 110 MiB | 9.3x | 90s | No |
| zip -9 (per-file) | 200 MiB | 5.1x | 30s | Yes |
| 7z -mx5 (solid) | 95 MiB | 10.8x | 120s | Per solid block |
| 7z -mx9 (solid) | 85 MiB | 12.0x | 400s | Per solid block |

### When 7z Excels

$$\text{7z Advantage} = \text{Solid compression} + \text{BCJ filters} + \text{LZMA2}$$

Best for: software distribution, source code archives, backup of similar files.

Worst for: archives needing fast random access to individual files.

## 7. Summary of Functions by Type

| Formula | Math Type | Application |
|:---|:---|:---|
| $C(f_1 \| f_2 \| \ldots)$ | Concatenation | Solid compression |
| $\text{Filter}(\text{Filter}(\text{Input}))$ | Function composition | Filter chains |
| $2^{19}$ iterations | Exponential | Key derivation cost |
| $\lceil \frac{\text{Size}}{\text{Vol}} \rceil$ | Ceiling | Volume splitting |
| $2^n$ dictionary | Exponential | Memory scaling |
| $1 - \frac{C_{solid}}{C_{non}}$ | Ratio | Solid improvement |
