# The Mathematics of Bracket Matching -- Stacks, Grammars, and the Dyck Language

> *A string of brackets is valid if and only if it belongs to the Dyck language -- the simplest context-free language that is not regular. The stack is the minimal machine that recognizes it.*

---

## 1. The Dyck Language and Context-Free Grammars (Formal Languages)

### The Problem

Let $\Sigma = \{(, ), [, ], \{, \}\}$ be the alphabet of bracket characters. We want to decide membership in the language of all correctly nested bracket strings. This is the **Dyck language** $D_3$ (Dyck language over 3 bracket types).

### The Formula

The Dyck language $D_k$ over $k$ bracket types is generated by the context-free grammar:

$$S \to \epsilon \mid S \, S \mid (S) \mid [S] \mid \{S\}$$

where $\epsilon$ is the empty string. A string $w \in \Sigma^*$ is valid if and only if $w \in L(S)$, i.e., it can be derived from the start symbol $S$.

The **Chomsky-Schutzenberger theorem** states that every context-free language is the homomorphic image of the intersection of a Dyck language with a regular language. This makes the Dyck language a fundamental building block of formal language theory.

### Worked Examples

**Example 1:** The string "{[]}" has the derivation:

$$S \Rightarrow \{S\} \Rightarrow \{S \, S\} \Rightarrow \{[S] \, S\} \Rightarrow \{[\epsilon] \, \epsilon\} = \{[]\}$$

Wait -- let us be more careful. Actually:

$$S \Rightarrow \{S\} \Rightarrow \{[S]\} \Rightarrow \{[\epsilon]\} = \{[]\}$$

The string has a valid derivation, so it is in $D_3$.

**Example 2:** The string "([)]" has no valid derivation. After reading "([", the stack contains $[(, []$. The next character ")" requires the top of the stack to be "(", but it is "[". No production rule allows this cross-nesting.

## 2. Stack-Based Recognition -- Pushdown Automata (Automata Theory)

### The Problem

The Dyck language is context-free but not regular. By the pumping lemma for regular languages, no finite automaton can recognize it. We need a **pushdown automaton** (PDA) -- a finite automaton augmented with a stack.

### The Formula

The deterministic PDA for bracket matching operates as follows. Let the stack alphabet be $\Gamma = \{(, [, \{, \bot\}$ where $\bot$ is the bottom-of-stack marker. The transition function $\delta$ is:

$$\delta(q_0, c, \gamma) = \begin{cases}
(q_0, \text{push}(c)) & \text{if } c \in \{(, [, \{\} \\
(q_0, \text{pop}) & \text{if } (c, \gamma) \in \{(), (, [,]), \{,\}\} \\
(q_{\text{reject}}, \cdot) & \text{otherwise}
\end{cases}$$

The string is accepted if and only if after processing all characters, the stack contains only $\bot$ (i.e., the stack is empty) and we are in state $q_0$.

### Worked Examples

**Trace for "{[]}":**

| Step | Character | Stack (top right) | Action |
|------|-----------|-------------------|--------|
| 0 | -- | $\bot$ | Initial |
| 1 | `{` | $\bot \, \{$ | Push `{` |
| 2 | `[` | $\bot \, \{ \, [$ | Push `[` |
| 3 | `]` | $\bot \, \{$ | Pop `[` (matches `]`) |
| 4 | `}` | $\bot$ | Pop `{` (matches `}`) |

Stack is empty at end: **accept**.

**Trace for "([)]":**

| Step | Character | Stack (top right) | Action |
|------|-----------|-------------------|--------|
| 0 | -- | $\bot$ | Initial |
| 1 | `(` | $\bot \, ($ | Push `(` |
| 2 | `[` | $\bot \, ( \, [$ | Push `[` |
| 3 | `)` | $\bot \, ( \, [$ | Top is `[`, expected `(`: **reject** |

Mismatch at step 3: **reject**.

## 3. Counting Valid Bracket Strings -- Catalan Numbers (Combinatorics)

### The Problem

How many distinct valid bracket strings of length $2n$ exist when using a single bracket type? This count has deep connections to combinatorics.

### The Formula

The number of valid strings of $n$ pairs of a single bracket type is the $n$-th **Catalan number**:

$$C_n = \frac{1}{n+1}\binom{2n}{n} = \frac{(2n)!}{(n+1)! \cdot n!}$$

The first few values are: $C_0 = 1, C_1 = 1, C_2 = 2, C_3 = 5, C_4 = 14, C_5 = 42$.

For $k$ bracket types, the count of valid strings of length $2n$ is the **Fuss-Catalan number** generalization, but a simpler way to state it: the number of valid strings of length $2n$ using $k$ bracket types is:

$$C_n^{(k)} = \frac{k^n}{n+1}\binom{2n}{n}$$

### Worked Examples

**Single bracket type, $n = 3$ (strings of length 6):**

$$C_3 = \frac{1}{4}\binom{6}{3} = \frac{1}{4} \times 20 = 5$$

The five valid strings: `((()))`, `(()())`, `(())()`, `()(())`, `()()()`.

**Three bracket types, $n = 1$ (strings of length 2):**

$$C_1^{(3)} = \frac{3^1}{2}\binom{2}{1} = \frac{3}{2} \times 2 = 3$$

The three valid strings: `()`, `[]`, `{}`.

## 4. Amortized Analysis of Stack Operations (Complexity Theory)

### The Problem

The algorithm processes $n$ characters, performing at most one push or one pop per character. Is the $O(n)$ bound tight, and what is the amortized cost per operation?

### The Formula

Using the **aggregate method**: across all $n$ characters, the total number of pushes is at most $n/2$ (since only opening brackets are pushed), and the total number of pops is at most $n/2$ (since each pop corresponds to a prior push). Therefore:

$$T_{\text{total}} = \text{pushes} + \text{pops} + \text{comparisons} \le \frac{n}{2} + \frac{n}{2} + n = 2n$$

The amortized cost per character is:

$$T_{\text{amortized}} = \frac{T_{\text{total}}}{n} = \frac{2n}{n} = O(1)$$

Using the **potential method**: define the potential function $\Phi = |\text{stack}|$ (the number of elements on the stack). For a push operation, the actual cost is 1 and $\Delta\Phi = +1$, giving amortized cost $1 + 1 = 2$. For a pop operation, the actual cost is 1 (including comparison) and $\Delta\Phi = -1$, giving amortized cost $1 - 1 = 0$. The amortized cost is $O(1)$ per operation either way.

### Worked Examples

For the string `{[()]}` of length 6:

- 3 pushes (characters `{`, `[`, `(`) -- total push work: 3
- 3 pops with comparisons (characters `)`, `]`, `}`) -- total pop work: 3
- Total operations: 6 = $n$

For the string `()()()` of length 6:

- 3 pushes (characters `(`, `(`, `(`) -- total push work: 3
- 3 pops (characters `)`, `)`, `)`) -- total pop work: 3
- Total operations: 6 = $n$

Maximum stack depth varies (3 for `{[()]}`, 1 for `()()()`), but total work is always $O(n)$.

## 5. Space Optimality and the Irregular Language Argument (Lower Bounds)

### The Problem

Can we solve valid parentheses with less than $O(n)$ space? For a single bracket type, a simple counter suffices ($O(\log n)$ space). Does this extend to multiple bracket types?

### The Formula

For $k \ge 2$ bracket types, the language is **not regular** (by the pumping lemma), and any streaming algorithm requires $\Omega(n)$ space in the worst case. The argument:

Consider the family of strings $w_m = s_1 s_2 \cdots s_m$ where each $s_i \in \{(, [\}$. There are $2^m$ distinct prefixes, and for each prefix, the set of valid completions is different. By the Myhill-Nerode theorem, the number of equivalence classes (and hence the minimum number of DFA states) is at least $2^m$. Since a DFA with $q$ states requires $\Omega(\log q)$ space, any recognizer needs $\Omega(m) = \Omega(n)$ space.

For a single bracket type ($k = 1$), the language $\{(^n )^n : n \ge 0\}$ requires only a counter, giving $O(\log n)$ space. But this does not generalize because the counter cannot track the *order* of different bracket types.

### Worked Examples

The prefixes `([` and `[(` require different completions:
- `([` must be followed by `])` (not `)]`)
- `[(` must be followed by `)]` (not `])`)

A counter alone cannot distinguish these two states. The stack encoding is necessary and its worst-case depth of $n/2$ gives $\Theta(n)$ space, which is optimal for $k \ge 2$.

## Prerequisites

| Concept | Why It Matters |
|---------|---------------|
| Stack (LIFO) data structure | The core data structure: push for openers, pop-and-compare for closers |
| Context-free grammars | Valid bracket strings form the Dyck language, the canonical CFL |
| Pushdown automata | The theoretical machine that recognizes CFLs using a stack |
| Catalan numbers | Count the number of valid bracket strings of a given length |
| Pumping lemma | Proves that multiple bracket types require more than finite-state memory |

## Complexity

| Metric | Value |
|--------|-------|
| Time | $O(n)$ -- single pass through the string |
| Space | $O(n)$ -- stack depth up to $n/2$ in the worst case |
| Space (single bracket type) | $O(\log n)$ -- a counter suffices |
| Amortized cost per character | $O(1)$ -- each character triggers at most one push or pop |
