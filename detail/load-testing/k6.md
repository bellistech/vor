# The Mathematics of k6 — Virtual Users, Arrival Rates, and Performance Modeling

> *k6's execution model spans closed-loop (virtual user) and open-loop (arrival rate) paradigms. The mathematics involves queuing theory, Little's Law, percentile estimation from streaming histograms, and threshold-based statistical acceptance testing.*

---

## 1. Execution Models (Queuing Theory)

### Closed Model (constant-vus)

In a closed model, $N$ virtual users (VUs) loop continuously. Each iteration takes time $T_{iter} = T_{req} + T_{think}$:

$$\text{Throughput} = \frac{N}{E[T_{iter}]} = \frac{N}{E[T_{req}] + T_{think}}$$

This is a direct application of **Little's Law**:

$$L = \lambda W \implies N = \text{RPS} \times E[T_{iter}]$$

| VUs ($N$) | Avg Response ($T_{req}$) | Think Time | Throughput (RPS) |
|:---:|:---:|:---:|:---:|
| 10 | 100 ms | 1 s | 9.1 |
| 50 | 100 ms | 1 s | 45.5 |
| 50 | 500 ms | 1 s | 33.3 |
| 100 | 200 ms | 0 s | 500.0 |
| 100 | 200 ms | 1 s | 83.3 |

### The Saturation Problem

As server response time increases under load, closed-model throughput self-limits:

$$\text{RPS}(N) = \frac{N}{T_{req}(N) + T_{think}}$$

When $T_{req}(N)$ grows due to server saturation:

$$\lim_{T_{req} \to \infty} \text{RPS}(N) = 0$$

This masks performance problems because the load generator slows down with the server.

### Open Model (constant-arrival-rate)

In an open model, requests arrive at a fixed rate $\lambda$ regardless of response time:

$$\text{RPS} = \lambda \quad (\text{independent of } T_{req})$$

Required VUs to sustain the rate:

$$N_{required} = \lambda \times E[T_{req}]$$

$$N_{max} = \lambda \times T_{req,max} \quad (\text{worst case})$$

| Target RPS ($\lambda$) | Avg Response | Min VUs | Recommended maxVUs |
|:---:|:---:|:---:|:---:|
| 100 | 100 ms | 10 | 50 |
| 100 | 500 ms | 50 | 150 |
| 500 | 200 ms | 100 | 300 |
| 1000 | 100 ms | 100 | 500 |
| 1000 | 1 s | 1000 | 2000 |

## 2. Ramping Stages — Load Profile Integration

### Linear Interpolation

During a ramp from $V_0$ to $V_1$ over duration $D$:

$$V(t) = V_0 + (V_1 - V_0) \cdot \frac{t}{D}$$

Total VU-seconds (area under the curve):

$$\text{VU-seconds} = \int_0^D V(t)\,dt = D \cdot \frac{V_0 + V_1}{2}$$

### Multi-Stage Total Load

For $k$ stages with start/end VUs $(V_{i-1}, V_i)$ and duration $D_i$:

$$\text{Total VU-seconds} = \sum_{i=1}^{k} D_i \cdot \frac{V_{i-1} + V_i}{2}$$

$$\text{Total Iterations} \approx \frac{\text{Total VU-seconds}}{E[T_{iter}]}$$

### Example Ramp Profile

| Stage | Duration | Start VUs | End VUs | VU-seconds |
|:---|:---:|:---:|:---:|:---:|
| Ramp up | 2 min | 0 | 50 | 3,000 |
| Hold | 5 min | 50 | 50 | 15,000 |
| Spike | 1 min | 50 | 200 | 7,500 |
| Hold | 3 min | 200 | 200 | 36,000 |
| Ramp down | 2 min | 200 | 0 | 12,000 |
| **Total** | **13 min** | | | **73,500** |

At $E[T_{iter}] = 1.5s$: Total iterations $\approx 49,000$.

## 3. Percentile Estimation (HDR Histogram)

### Streaming Percentile Computation

k6 uses a histogram-based approach to compute percentiles over millions of observations without storing every value.

For $n$ observations sorted as $x_{(1)} \leq x_{(2)} \leq \ldots \leq x_{(n)}$, the $p$-th percentile:

$$P_p = x_{(\lceil p \cdot n \rceil)}$$

### Precision and Bucket Count

With a histogram of $B$ buckets spanning range $[x_{min}, x_{max}]$ using logarithmic spacing:

$$B = \frac{\log(x_{max}/x_{min})}{\log(1 + \epsilon)} \quad \text{where } \epsilon \text{ is relative error}$$

| Range | Precision ($\epsilon$) | Buckets | Memory |
|:---|:---:|:---:|:---:|
| 1ms - 60s | 1% | 856 | ~7 KiB |
| 1ms - 60s | 0.1% | 8,560 | ~67 KiB |
| 1us - 3600s | 1% | 1,370 | ~11 KiB |

### Threshold Math

A threshold like `p(95)<500` tests:

$$H_0: P_{95} \geq 500\text{ms} \quad H_1: P_{95} < 500\text{ms}$$

k6 evaluates this as a simple comparison at test end. The confidence depends on sample size:

$$\text{SE}(P_p) \approx \frac{1}{f(P_p)} \sqrt{\frac{p(1-p)}{n}}$$

where $f(P_p)$ is the probability density at the percentile value.

| Samples ($n$) | Relative SE of p95 | Practical |
|:---:|:---:|:---|
| 100 | ~22% | Very noisy |
| 1,000 | ~7% | Marginal |
| 10,000 | ~2.2% | Good |
| 100,000 | ~0.7% | Excellent |

## 4. Check Pass Rate (Binomial Model)

### Pass Rate Threshold

For a check with threshold `rate>0.99`:

$$\hat{p} = \frac{\text{passed}}{n}, \quad \text{Threshold: } \hat{p} > 0.99$$

Confidence interval for the true pass rate:

$$\hat{p} \pm z_{\alpha/2} \sqrt{\frac{\hat{p}(1-\hat{p})}{n}}$$

### Required Sample Size

To detect a true rate of $p = 0.98$ versus threshold $p_0 = 0.99$ with power $\beta = 0.80$:

$$n = \left(\frac{z_\alpha \sqrt{p_0(1-p_0)} + z_\beta \sqrt{p(1-p)}}{p_0 - p}\right)^2$$

$$n = \left(\frac{1.645\sqrt{0.0099} + 0.842\sqrt{0.0196}}{0.01}\right)^2 \approx 6,800$$

## 5. Throughput Capacity Planning

### Maximum Throughput

Server capacity $C$ (max RPS) depends on:

$$C = \min\left(\frac{\text{CPU cores}}{T_{cpu}}, \frac{\text{Connections}}{T_{conn}}, \frac{\text{Bandwidth}}{S_{response}}\right)$$

### Response Time Under Load (M/M/c Queue)

For $c$ server workers, arrival rate $\lambda$, service rate $\mu$:

$$\rho = \frac{\lambda}{c \cdot \mu} \quad (\text{utilization})$$

Average response time:

$$E[T] = \frac{1}{\mu} + \frac{P_q}{c \cdot \mu - \lambda} \cdot \frac{1}{\mu}$$

| Utilization ($\rho$) | Response Time Multiplier |
|:---:|:---:|
| 50% | 1.3x baseline |
| 70% | 2.0x baseline |
| 80% | 3.0x baseline |
| 90% | 6.0x baseline |
| 95% | 12x baseline |
| 99% | 50x baseline |

## 6. Data-Driven Testing — Memory Model

### SharedArray Memory

Without SharedArray, each VU copies the data:

$$M_{naive} = N_{VUs} \times S_{data}$$

With SharedArray:

$$M_{shared} = S_{data} + N_{VUs} \times S_{reference}$$

| Dataset Size | 100 VUs (naive) | 100 VUs (shared) | Savings |
|:---:|:---:|:---:|:---:|
| 1 MiB | 100 MiB | ~1 MiB | 99% |
| 10 MiB | 1 GiB | ~10 MiB | 99% |
| 100 MiB | 10 GiB | ~100 MiB | 99% |

## Prerequisites

- Queuing theory (Little's Law, M/M/c queues)
- Probability distributions (geometric, binomial)
- Percentile estimation and histograms
- Statistical hypothesis testing
- Integration for area-under-curve calculations

## Complexity

| Operation | Time | Space |
|:---|:---|:---|
| VU iteration | $O(1)$ per request | $O(1)$ per VU |
| Percentile computation | $O(1)$ histogram insert | $O(B)$ histogram buckets |
| Threshold evaluation | $O(t)$ thresholds | $O(1)$ per threshold |
| SharedArray lookup | $O(1)$ index | $O(n)$ shared, not per-VU |
| Check evaluation | $O(c)$ checks per iteration | $O(1)$ counter per check |
| Summary generation | $O(m)$ metrics | $O(m \times B)$ histograms |
