# The Mathematics of S3 — Storage Economics, Throughput Limits, and Consistency Model

> *S3 provides virtually unlimited object storage with a pay-per-use model. The math covers storage cost optimization across tiers, request pricing, multipart upload partitioning, consistency guarantees, and the combinatorics of lifecycle policy transitions.*

## 1. Storage Cost Optimization (Economics)

### Monthly Cost per Storage Class

$$C_{\text{monthly}} = S \times P_{\text{storage}} + R_{\text{get}} \times P_{\text{get}} + R_{\text{put}} \times P_{\text{put}} + D_{\text{retrieval}} \times P_{\text{retrieval}}$$

### Break-Even Analysis (Standard vs Standard-IA)

$$C_{\text{std}} = S \times 0.023 + R \times 0.0004/1000$$

Break-even when $C_{\text{std}} = C_{\text{ia}}$:

$$\text{Access frequency threshold} \approx \frac{0.0105 \times S}{0.0006/1000 + 0.01 \times f}$$

## 2. Multipart Upload Optimization (Partitioning)

### Optimal Part Size

AWS constraints: min 5 MB/part, max 10,000 parts, max 5 TB object.

$$\text{Part size} = \max\left(5 \text{ MB}, \left\lceil \frac{S_{\text{file}}}{10000} \right\rceil\right)$$

$$\text{Parts} = \left\lceil \frac{S_{\text{file}}}{\text{Part size}} \right\rceil$$

## 3. Request Rate and Partitioning (Throughput)

### S3 Performance Limits

Per prefix:
- 5,500 GET/HEAD requests per second
- 3,500 PUT/COPY/POST/DELETE requests per second

### Partition Strategy

Distribute keys across prefixes to increase aggregate throughput:

$$\text{Aggregate GETs/sec} = P \times 5500$$

$$\text{Aggregate PUTs/sec} = P \times 3500$$

where $P$ = number of distinct prefixes used.

| Prefixes | Max GETs/sec | Max PUTs/sec |
|:---:|:---:|:---:|
| 1 | 5,500 | 3,500 |
| 4 | 22,000 | 14,000 |
| 16 | 88,000 | 56,000 |
| 100 | 550,000 | 350,000 |

### Key Distribution

Using hash-based prefixes:

$$\text{key} = \text{hex(hash(id))[0:2]}/\text{id}/\text{filename}$$

This creates 256 prefixes (00-ff), yielding:

$$\text{Max throughput} = 256 \times 5500 = 1,408,000 \text{ GETs/sec}$$

## 4. Consistency Model (Distributed Systems)

### Strong Read-After-Write Consistency (since Dec 2020)

$$\text{PUT}(k, v) \rightarrow \text{acknowledge} \rightarrow \text{GET}(k) = v \quad \text{(guaranteed)}$$

$$\text{DELETE}(k) \rightarrow \text{acknowledge} \rightarrow \text{GET}(k) = 404 \quad \text{(guaranteed)}$$

### List Consistency

$$\text{PUT}(k) \rightarrow \text{LIST(prefix)} \ni k \quad \text{(eventually, usually immediate)}$$

### Conditional Operations

S3 supports conditional writes (since 2024):

$$\text{PUT}(k, v) \text{ IF-NONE-MATCH } * \quad \text{(write only if key doesn't exist)}$$

This enables safe concurrent writes without external locking.

## 6. Data Durability and Availability (Reliability)

### Durability Model

S3 Standard provides 99.999999999% (11 nines) durability:

$$P(\text{loss}) = 10^{-11} \text{ per object per year}$$

### Expected Object Loss

$$E[\text{lost objects}] = N \times P(\text{loss})$$

| Objects Stored | Expected Loss/Year |
|:---:|:---:|
| 1 million | 0.00001 |
| 1 billion | 0.01 |
| 100 billion | 1 |
| 1 trillion | 10 |

### Availability SLA

| Storage Class | Availability SLA | Downtime/Year |
|:---|:---:|:---:|
| Standard | 99.99% | 52.6 min |
| Standard-IA | 99.9% | 8.77 hours |
| One Zone-IA | 99.5% | 1.83 days |
| Glacier | 99.99% | 52.6 min |

### Replication

S3 stores objects across at least 3 AZs. For AZ failure probability $p$:

$$P(\text{data loss}) = p^3$$

| AZ Failure Rate | 3-AZ Durability |
|:---:|:---:|
| 0.01 | 0.000001 |
| 0.001 | 0.000000001 |
| 0.0001 | $10^{-12}$ |
