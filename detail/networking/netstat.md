# netstat Deep Dive — Theory & Internals

> *netstat reads /proc/net to display socket tables, routing information, and interface statistics. While largely superseded by ss and ip, understanding netstat's output means understanding the kernel's view of network state — connection counts, socket exhaustion, and interface error rates.*

---

## 1. Connection State Counting

### The /proc/net/tcp Structure

netstat reads `/proc/net/tcp` (and tcp6, udp, udp6) — text files generated by the kernel.

$$T_{parse} = O(N) \quad \text{where } N = \text{total socket count}$$

### State Summary (`-s` flag)

$$N_{total} = \sum_{s \in \text{states}} N_s$$

### Connection State Health Metrics

| Metric | Formula | Healthy Value |
|:---|:---|:---|
| Established ratio | $N_{ESTAB} / N_{total}$ | > 80% |
| TIME-WAIT ratio | $N_{TW} / N_{total}$ | < 30% |
| CLOSE-WAIT count | $N_{CW}$ | 0 (any = application bug) |
| SYN-RECV count | $N_{SR}$ | < 50 (> 500 = SYN flood) |

### TIME-WAIT Ephemeral Port Exhaustion

$$P_{exhausted} = \frac{N_{TW}}{P_{range}}$$

Where $P_{range} = \text{ip\_local\_port\_range}$ (default: 32768-60999 = 28,232 ports).

| TIME-WAIT Count | Port Usage | Risk |
|:---:|:---:|:---|
| 1,000 | 3.5% | None |
| 10,000 | 35.4% | Monitor |
| 20,000 | 70.8% | Warning |
| 28,000 | 99.2% | Critical — connections fail |

## 2. Interface Statistics (`-i` flag)

### Error Rate Calculations

$$\text{Error rate} = \frac{E_{errors}}{P_{total}} \times 100\%$$

### Key Counters

| Counter | Meaning | Concern Threshold |
|:---|:---|:---:|
| RX-OK | Packets received successfully | N/A (reference) |
| RX-ERR | Receive errors | > 0.01% |
| RX-DRP | Packets dropped (buffer overflow) | > 0.001% |
| RX-OVR | Ring buffer overruns | Any > 0 |
| TX-OK | Packets transmitted successfully | N/A |
| TX-ERR | Transmit errors | > 0.01% |
| TX-DRP | Transmit drops | > 0.001% |
| TX-OVR | Transmit overruns | Any > 0 |

### Drop Rate Trending

$$R_{drop} = \frac{\Delta \text{DRP}}{\Delta t}$$

$$\text{Drop trend} = R_{drop}(t) - R_{drop}(t - \Delta t)$$

Increasing drop rate = growing congestion or buffer exhaustion.

### Utilization from Counters

$$U = \frac{\Delta \text{bytes} \times 8}{\Delta t \times BW_{link}} \times 100\%$$

| Interface | Bytes in 60 sec | Utilization (1 Gbps) |
|:---:|:---:|:---:|
| 7,500,000,000 | 7.5 GB | 100% |
| 3,750,000,000 | 3.75 GB | 50% |
| 750,000,000 | 750 MB | 10% |

## 3. Routing Table (`-r` flag)

### Route Selection

netstat displays the kernel routing table. For each destination, the kernel selects:

$$\text{Route} = \arg\max_{r \in \text{routes}} \text{prefix\_length}(r) \quad \text{s.t. } \text{dest} \in r.\text{network}$$

Longest prefix match — the most specific route wins.

### Route Metrics

| Field | Meaning | Impact |
|:---|:---|:---|
| Destination | Network prefix | Match target |
| Gateway | Next hop | Forwarding address |
| Genmask | Subnet mask | Prefix length |
| Metric | Route preference | Lower = preferred |
| Ref | Reference count | In-use indicator |
| Use | Lookup count | Popularity |

### Default Route

$$0.0.0.0/0 \quad \text{(matches everything, lowest priority)}$$

## 4. Protocol Statistics (`-s` flag)

### Key TCP Statistics

| Statistic | Healthy Range | Concern |
|:---|:---|:---|
| segments retransmited / segments sent | < 1% | > 5% = significant loss |
| connections reset | Low relative to established | High = application crashes |
| passive opens / active opens | Server ratio | Varies by workload |
| failed connection attempts | < 1% of attempts | Connection refused or timeout |

### Retransmission Rate

$$R_{retrans} = \frac{\text{segments retransmitted}}{\text{segments sent}} \times 100\%$$

| Rate | Quality |
|:---:|:---|
| < 0.1% | Excellent |
| 0.1-1% | Acceptable |
| 1-5% | Degraded |
| > 5% | Severe loss |

### UDP Statistics

$$\text{Buffer overflow rate} = \frac{\text{receive buffer errors}}{\text{datagrams received}} \times 100\%$$

UDP buffer errors = application not reading fast enough.

## 5. Performance: netstat vs ss

### Data Source Comparison

| Aspect | netstat | ss |
|:---|:---|:---|
| Data source | /proc/net/* (text files) | Netlink (binary protocol) |
| Parsing | User-space text parsing | Kernel-side filtering |
| Kernel locking | Longer lock hold | Shorter lock hold |
| Speed (10K sockets) | ~1 sec | ~0.1 sec |
| Speed (100K sockets) | ~10 sec | ~1 sec |
| Memory usage | Higher (full text buffer) | Lower (binary) |

### Scaling Formula

$$T_{netstat} \approx a \times N + b \quad \text{(linear, with large constant)}$$

$$T_{ss} \approx c \times N + d \quad \text{(linear, smaller constant, kernel filtering)}$$

Where $a >> c$ due to text parsing overhead.

### When netstat Causes Problems

At high socket counts, netstat holds kernel locks while reading /proc/net/tcp:

$$T_{lock} = O(N) \quad \text{(entire socket table is locked)}$$

On a server with 500K connections, netstat can cause application stalls of several seconds due to lock contention.

## 6. Listening Service Enumeration (`-tlnp`)

### Service Discovery

$$N_{listening} = |\{s : s.\text{state} = \text{LISTEN}\}|$$

### Port Conflict Detection

$$\text{Conflict} = \exists \; s_1, s_2 : s_1.\text{port} = s_2.\text{port} \wedge s_1.\text{addr} \cap s_2.\text{addr} \neq \emptyset$$

Port conflicts are detected when two processes try to bind the same (address, port):
- `0.0.0.0:80` and `192.168.1.1:80` may conflict depending on `SO_REUSEADDR`.
- `0.0.0.0:80` and `0.0.0.0:80` always conflict (unless `SO_REUSEPORT`).

## 7. Summary of Formulas

| Formula | Math Type | Application |
|:---|:---|:---|
| $N_{TW} / P_{range}$ | Ratio | Port exhaustion risk |
| $E / P \times 100$ | Percentage | Interface error rate |
| $\Delta \text{bytes} \times 8 / (\Delta t \times BW)$ | Utilization | Link usage |
| $\text{retrans} / \text{sent}$ | Ratio | TCP retransmission rate |
| $O(N)$ parsing | Complexity | netstat scaling |
| $\arg\max(\text{prefix\_len})$ | Longest match | Route selection |

## Prerequisites

- socket state machines, routing table lookups, process identification

*netstat is the legacy tool that everyone still uses because it's everywhere. On modern systems with tens of thousands of connections, ss is 10x faster, but netstat's output format is burned into the muscle memory of every sysadmin who's ever debugged a connection leak at 3 AM.*
